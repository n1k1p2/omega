## 2. Типографика

### Шрифтовая пара (Google Fonts, полная кириллица)

- **Заголовочный: «Manrope»** — нет, слишком нейтральный для «характерного». Пересмотрено ниже — используем контрастную антикву для тона «фабрика с историей + премиум».

**Заголовочный — `Unbounded`** (Google Fonts, кириллица подтверждена в наборе). Современный широкий гротеск с нативной кириллицей — та самая пластика tech-референса gigacode, смелая и запоминающаяся. Веса: 500/600 для h3–h4, 600/700 для h1–h2. РЕШЕНИЕ ЗАКАЗЧИКА (10.07.2026): антиква (бывш. Playfair) ЗАПРЕЩЕНА — заказчик воспринял её как «Times New Roman». Никаких serif-шрифтов на сайте.
  - Начертания: `Unbounded: wght@500;600;700;800`, обычный + italic (italic — для акцентных фраз-цитат в отзывах и УТП-инсертов).

- **Текстовый — `Manrope`** (Google Fonts, полная кириллица). Нейтральный гуманистический гротеск, отличная читаемость на мелких кеглях (характеристики, описание, формы), современный «tech» подтон, который даёт нужную перекличку с пластикой референса gigacode, не выглядя холодно. Веса: 400 (Regular) — body, 500 (Medium) — акцентный текст/лейблы, 600 (Semibold) — кнопки/цены/навигация, 700 (Bold) — крупные цифры счётчиков.
  - Правка С7: `Manrope` не имеет истинных табличных цифр как отдельного OpenType-фичи, поэтому все анимируемые/пересчитываемые числа (счётчики статистики, номер заказа) обязаны идти с явным CSS `font-variant-numeric: tabular-nums`, чтобы ширина цифр не «дёргалась» при смене значения. `Unbounded` для таких целей не используется вообще (только для статичной, уже финальной цены — она не анимируется, см. правку К2).

- Правка С9: моно-шрифт `JetBrains Mono` из спецификации **исключён**. Причина: поля «артикул» нет ни в products.json, ни в api-contract.md; единственный код, который реально показываем — `Код: OM-{id}` (см. правку 4.8/5.3 ниже) и номер заказа `OM-000001`, которые прекрасно читаются в Manrope с `tabular-nums`. Убираем подключение третьего шрифта → минус лишний файл шрифта, чуть лучше LCP/меньше блокирующих запросов. Итог: **2 шрифта на сайте — Unbounded + Manrope**.

## 3. Токены: радиусы, тени, отступы, контейнер, сетка

### Радиусы
- `--radius-sm`: 8px — инпуты, чипсы, свотчи (квадратные)
- `--radius-md`: 14px — кнопки, бейджи, поля форм
- `--radius-lg`: 20px — карточки товаров, карточки блоков
- `--radius-xl`: 28px — крупные модули (hero-карточка, модалки на десктопе)
- `--radius-full`: 999px — круглые кнопки (иконки избранного/сравнения), свотчи цвета, аватары отзывов, pill-бейджи

Правка В8 (тач-таргеты ≥44×44px, §7 CRO-чеклиста): визуальный размер круглых иконок-кнопок (избранное) и свотчей может оставаться小 (32–36px визуал, 16px свотч), но **хит-зона клика/тапа обязана быть ≥44×44px** за счёт прозрачного padding вокруг визуального элемента — задавать через `min-width/min-height: 44px` на кликабельном родителе, не на самой иконке. Расстояние между соседними хит-зонами ≥8px. На мобильном листинге (карточка 4.1) свотчи цвета — некликабельные статичные индикаторы (см. правку В3), поэтому требование 44px к ним не применяется — им не нужна хит-зона.

### Тени (тёплые, не серые — с оттенком foreground/accent, чтобы не выглядели «холодным материал-дизайном»)
- `--shadow-sm`: `0 1px 2px rgba(43,36,32,0.06), 0 1px 1px rgba(43,36,32,0.04)` — инпуты, чипы
- `--shadow-md`: `0 4px 16px rgba(43,36,32,0.08), 0 2px 6px rgba(43,36,32,0.05)` — карточки товара в покое
- `--shadow-lg`: `0 16px 40px rgba(43,36,32,0.14), 0 6px 16px rgba(43,36,32,0.08)` — карточка товара hover/поднятие, модалки
- `--shadow-glow-amber`: `0 0 0 1px rgba(156,122,46,0.25), 0 8px 32px rgba(217,165,74,0.25)` — активный/выбранный элемент (выбранный размер, активная вкладка квиза, hover карточки категории)

### 4.8 Слайдер галереи (PDP)

- Desktop: миниатюры вертикальной колонкой слева (64×64px, `radius-sm`, активная — обводка `primary` 2px), основное фото справа (соотношение 1:1, `radius-lg`, фон `surface`), стрелки навигации появляются на hover области фото (круглые 44px, `surface` 90% + `shadow-md`).
- Порядок слайдов: 1) интерьерное/lifestyle или лучший ракурсный рендер, 2) деталь (фурнитура/ткань крупный план), 3) механизм/конструкция (если применимо), 4–6) рендеры на белом с разных углов.
- Зум: клик открывает лайтбокс с pinch-to-zoom на мобильном, курсор-лупа на desktop при hover (CSS `background-position` по координатам курсора).
- **Синхронизация со свотчем — убрана как основной сценарий** (правка В3): в API нет маппинга цвет→фото ни в `images`, ни где-либо ещё, значит условие «если фото есть» срабатывать не будет практически никогда — на деле это 100%-hedge, а не edge case. Дефолтное и единственное поведение: при выборе цвета/ткани в блоке покупки галерея НЕ переключается, под свотчами/чипсами постоянно показывается статичная подпись `Small` `foreground-muted` «Фото может отличаться по цвету ткани и обивки». ### 4.9 Аккордеон

- Заголовок секции — H4 Manrope 700, высота строки 56px, иконка `+`/`−` справа (поворот 45°→0° на анимации, 200ms), разделитель снизу `border` 1px.
- Контент разворачивается через `grid-template-rows: 0fr → 1fr` (плавно, без скачков высоты, дружелюбно к CLS), 300ms ease-in-out. **Осознанное исключение из правила §6 «анимировать только transform/opacity»** (правка С6): `grid-template-rows` технически layout-анимация, но применяется точечно — только по явному клику пользователя, локально в пределах одного блока, не в scroll-обработчике и не массово на странице одновременно — поэтому не создаёт риска для скролл-перформанса/CLS, в отличие от hover/scroll-reveal анимаций, где правило transform/opacity обязательно без исключений.
- Используется на PDP: «Описание», «Характеристики» (таблица key-value из `features`), «Доставка и оплата», «Отзывы» — на desktop можно рендерить как табы (горизонтальные), на mobile — всегда аккордеон.

### 5.2 Каталог категории

**Правка К1 (критично) — фильтры и сортировка переписаны под реальный API-контракт.** Контракт поддерживает только `category, search, ordering (price/-price/name/-created_at/popular), is_bestseller, is_new, min_price, max_price`. Ни `size`, ни `color`, ни `mechanism`, ни `in_stock`-фильтра, ни фасетных счётчиков API не отдаёт. До расширения контракта — v1 ограничен ценой и честной сортировкой.

1. Хлебные крошки + H1 категории («Кровати») + короткое описание категории (1 строка) + счётчик «27 моделей».
2. Панель управления (sticky под хедером при скролле): слева — кнопка фильтров (открывает боковую панель desktop / bottom sheet mobile) с счётчиком активных фильтров, по центру/справа — сортировка (dropdown: **«По популярности» (`ordering=popular`) / «Сначала дешёвые» (`price`) / «Сначала дорогие» (`-price`) / «Новинки» (`-created_at` + фильтр `is_new=true`)** — пункт «По рейтингу» убран, в `ordering` такого значения нет), переключатель плотности сетки (иконки 4/3/2 в ряд, desktop only).
3. **Фильтры v1** (боковая панель desktop, ширина 280px, sticky): **только цена** — двойной слайдер + инпуты от/до (`min_price`/`max_price`), быстрые чипсы бюджета «до 30 000 ₽», «30 000–60 000 ₽», «от 60 000 ₽»; опционально чипсы «Хиты продаж» (`is_bestseller=true`) и «Новинки» (`is_new=true`) — оба поля есть в контракте. **Убраны из v1**: фильтр по размеру спального места, материалу/цвету, механизму трансформации, наличию — ни одного из этих параметров API не принимает, счётчик «(N) товаров останется» у чекбоксов убран целиком (требует фасетного endpoint, которого нет). Если фильтры по цвету/размеру нужны бизнесу — это отдельная задача на расширение `api-contract.md` (`color=`, `size=` — сидер уже парсит эти значения из `description`, добавить параметры в `ProductFilter` и вернуться к полной версии фильтров) до старта фронтенд-разработки, не после.

Порядок строго по CRO-чеклисту:

2. **Верхний блок** (2 колонки desktop 55/45, стек mobile): слева — галерея (4.8), справа — блок покупки:

### 5.10 Сравнение товаров — решение по правке В7

Иконки «Сравнить» стояли в хедере и на каждой карточке товара, но макета страницы `/compare` не было, а бриф обещал сравнительную таблицу линеек (Классика vs Престиж vs Империал) для сегмента E. Решение: **иконка «Сравнить» убрана из хедера и с карточки листинга в v1** (см. правки 4.1 и 4.5) — доводить до релиза недоделанный, нигде не специфицированный флоу хуже, чем не начинать его.

Взамен — специфицируется то, что реально закрывает потребность сегмента E без нового флоу состояния «добавить в сравнение / N выбрано»:
1. На PDP, в блоке «Похожие модели» (п.6 раздела 5.3), добавляется мини-таблица «Сравнить с похожими» — 3–4 колонки (текущий товар + `related`), строки: фото, цена, габариты (из `features`), материал/цвет, кнопка «Подробнее»/«В корзину» в каждой колонке. Строится из уже загруженных данных (`related` в полной карточке товара), без нового состояния «список сравнения» и без отдельного роута.
2. На странице категории (5.2), под сеткой товаров — статичный блок «Линейки Омега» с 3 карточками (Классика / Престиж / Империал, если такое деление прослеживается в `name`/`description` конкретной категории) и таблицей 4–5 строк (цена от, габариты, материал, особенность) — контент курируется вручную по существующим товарам, не по новому API.
Если в будущем спонсируется полноценная страница `/compare` с localStorage-списком — заводить как отдельную задачу с собственным макетом и `wishlist`-подобным состоянием.

## 6. Анимации — направление (детали дорабатывает моушн-дизайнер)

Перенос пластики референса gigacode на мебельную эстетику: та же плавность и «дороговизна» движения, но тёплая палитра свечений (янтарь/латунь вместо неона/техно-синего), и все декоративные эффекты подчинены правилу «не мешать LCP/CLS» из CRO-чеклиста.

### Общие принципы
- Все анимации — только `transform` и `opacity` (GPU-композиция), никогда `top/left/width/height/margin` в scroll-обработчиках. Единственное осознанное исключение — аккордеон `grid-template-rows` (см. правку С6 в 4.9), допустимое только по клику, локально.
- Easing по умолчанию: `cubic-bezier(0.22, 1, 0.36, 1)` («ease-out-expo-ish») для появления элементов — придаёт «мягкое торможение», ощущение качества.
- Длительности: микровзаимодействия (hover кнопки/карточки) 150–250ms; появление блоков 400–600ms; крупные декоративные (параллакс фото, счётчики) 1000–1800ms.
- `prefers-reduced-motion: reduce` — отключить весь parallax/marquee/decorative-glow, оставить только мгновенное fade (150ms) для смены состояний.
- Не более 2–3 одновременных scroll-triggered анимаций в вьюпорте (ограничение по производительности на слабых Android, как указано в CRO-чеклисте).
- **Правило В11 (обязательное, критично для LCP/CLS):** ни один блок не может быть невидимым (`opacity: 0`) по умолчанию в SSR-разметке. Все стартовые состояния scroll-reveal/hero-stagger анимаций (`opacity: 0`, `translateY(24px)` и т.п.) применяются ТОЛЬКО через модификатор-класс на `<html>` или общем враппере (например `.js-ready`, проставляется скриптом после гидрации/после первого `requestAnimationFrame`). Без JS или при его сбое весь контент, включая потенциальный LCP-элемент (hero H1 при desktop-сплите 55/45, где текст может быть крупнее фото на медленном канале), остаётся видимым сразу из чистого HTML/CSS. Reveal-анимации — прогрессивное улучшение поверх уже видимого контента, а не условие его появления. Это правило применяется одинаково к hero, scroll-reveal секций 4/5/6/7 и любым другим появлениям.

### Scroll-reveal (появление блоков при скролле)
- Паттерн: `opacity: 0 → 1` + `translateY(24px → 0)`, триггер — `IntersectionObserver` при пересечении 15–20% элемента с viewport, `threshold: 0.15`, срабатывает один раз (`once: true`).
- Для секций с несколькими карточками (Почему Омега, Хиты продаж, категории) — stagger-задержка 80–100ms между соседними карточками.
- Заголовки секций (H2) появляются чуть раньше контента (offset -50ms), создавая ощущение «сначала название, потом наполнение».

### Hero-анимация
- При загрузке: последовательный stagger caption → H1 (по словам или целиком) → subtitle → CTA-кнопки → соцдоказательство, шаг 80–120ms, начальное смещение `translateY(16px)`.
- Фоновое фото: мягкий `scale(1.05 → 1)` за 1200ms при первой загрузке (создаёт ощущение «кино», не связано с LCP-приоритетом самой отдачи файла).
- Декоративное свечение `gradient-brass-glow` за заголовком — статичное позиционирование, лёгкая пульсация opacity 0.6↔0.85 на 4s loop (очень тонкая, не должна отвлекать).
- При скролле вниз — параллакс hero-фото на 0.3x скорости скролла (создаёт глубину, отключается при `prefers-reduced-motion`).

### Marquee-лента
- Чистый CSS `@keyframes marquee { from: translateX(0); to: translateX(-50%) }` на удвоенном (продублированном) контенте, `will-change: transform`, `animation: marquee 30s linear infinite`.
- Пауза на `:hover` (desktop) через `animation-play-state: paused`.
- Ничего не завязано на JS/scroll — гарантированно не влияет на INP.

### Hover-пластика карточек товара
- Подъём карточки `translateY(-4px)` + усиление тени (`shadow-md → shadow-lg`), 220ms.
- Кросс-фейд фото на второй кадр, 400ms, синхронно с лёгким `scale(1.0 → 1.03)` фото внутри контейнера (контейнер `overflow: hidden`, чтобы не задевать соседние карточки).
- Кнопка «Заказать» — на desktop меняет визуальный вес (из приглушённой в полноценную primary) с fade 200ms, чтобы в состоянии покоя сетка каталога не «кричала» 24 кнопками одновременно, а фокусировалась на фото/цене.
- Иконки быстрых действий — fade-in 150ms + лёгкий `scale(0.9→1)`.

### Анимированные счётчики
- Числа считаются от 0 до целевого значения за 1400–1800ms, easing `easeOutExpo`, запуск через `IntersectionObserver` один раз при входе в viewport.
- Опционально — лёгкая дрожь/тик последней цифры (визуальный акцент завершения), но не обязательна.
- Подложка-свечение за иконкой счётчика (`gradient-brass-glow`) появляется с задержкой 200ms после начала счёта числа, создавая слоистость эффекта.

### Параллакс
- Максимум 2 слоя на секцию: (1) основное изображение 0.6–0.7x скорости скролла, (2) декоративный слой (частицы/линии/текстура дерева) 0.3–0.4x скорости — оба через `transform: translate3d`, обновление позиции через `requestAnimationFrame`, не на каждый scroll-event напрямую (throttle).
- Используется в блоках: hero (фон), «Из чего сделана мебель Омега» (фото среза дерева), «География доставки» (карта).

### Микровзаимодействия
- Свотчи цвета/ткани: при hover — лёгкий `scale(1.15)` + тонкая обводка `accent`, при выборе — пульс `box-shadow` один раз (200ms) как тактильное подтверждение.
- Чипсы размера/фильтра: при выборе — фон плавно заливается `gradient-amber` (200ms), текст становится белым.
- Живой пересчёт цены: число «прокатывается» покадрово от старого к новому значению (не мгновенная замена текста) — 300–400ms, эффект «одометра».
- Toast-уведомления: slide-in + fade, 250ms, иконка внутри может иметь микро-bounce (`scale(0.8→1.05→1)`, 350ms) для положительных действий («Добавлено в корзину»).
- Кнопки Primary L (hero/финальный CTA): едва заметный `scale(1.02)` на hover, тень усиливается — ощущение «кнопку хочется нажать».
- **Пульсация плавающей кнопки звонка (правка С5):** реализуется НЕ через анимацию `box-shadow` (это не композитный слой — браузер репейнтит кнопку и всё, что под ней, каждый цикл, каждые ~4с). Вместо этого — отдельный `::before`/`::after` псевдоэлемент того же радиуса, позиционированный абсолютно позади кнопки, анимируется через `transform: scale(1 → 1.4)` + `opacity: 0.5 → 0` в цикле 2s ease-out (эффект «расходящегося кольца», визуально читается так же, как пульсирующая тень, но целиком на GPU-композиции).

### Дополнительно (уникальные акценты для мебельного бренда, не из референса, но в той же пластике)
- **Эффект «сборки»** в блоке «Из чего сделана мебель Омега»: при scroll-reveal иконки механизмов/креплений можно анимировать лёгкой сборкой из смещённых частей в финальную позицию (translateX/Y схождение к центру, 500ms, stagger 60ms) — метафора «собрано добротно», связывает моушн с УТП инженерной надёжности.
- **Прогресс-линия квиза/калькулятора**: заполняется `gradient-amber`, с лёгким «свечением-кометой» на переднем крае полосы (второй, более яркий и размытый слой поверх заливки, двигается синхронно).

## 7. Мобильная версия — адаптация ключевых блоков

- **Header**: логотип по центру, гамбургер слева (открывает fullscreen-меню с категориями как аккордеон + топbar-инфо телефон/часы вверху меню), иконки корзина+избранное справа компактно. Поиск — отдельная иконка, разворачивается в fullwidth-оверлей поверх контента при тапе.
- **Hero**: фото на весь экран фоном (не сплит), текст поверх с `gradient-dark-fade` затемнением снизу-вверх для читаемости, H1 уменьшен до 38px, кнопки — стек (Primary сверху на всю ширину, Secondary под ней), высота hero — **`100svh`** минус высота хедера, но не более 640px (правка С3: `100vh` на мобильном скачет при появлении/скрытии адресной строки браузера, `100svh` — «small viewport height», стабильная минимальная высота — не нужно, чтобы CTA-кнопки уезжали за экран при первой отрисовке).
- **Marquee**: та же лента, но высота 44px, размер текста 11px, скорость чуть выше для компенсации меньшей ширины экрана (визуально одинаковый ритм).
- **Категории**: горизонтальный свайп-карусель вместо сетки (snap-scroll, `scroll-snap-type: x mandatory»), 1.15 карточки видно в кадре (намёк на следующую — «есть ещё»).
- **Хиты продаж / отзывы**: карусель свайпом, точки-индикатор снизу.
- **Квиз**: fullscreen-модалка (не инлайн-блок) при запуске с мобильного — меньше отвлекающих элементов, один вопрос на экран (см. сокращённые 2 шага, правка В13).
- **Подборщик компоновок** (было «Калькулятор»): вертикальный стек карточек компоновок с ценой сразу видной, без сложной визуальной схемы (см. правку В13 в 5.1 п.9).
- **Каталог**: см. п.3 токенов и 5.2 — 1 карточка на экран с горизонтальным свайпом фото, bottom sheet фильтров (в v1 — только цена, правка К1).
  - **Правка С4 (свайп-галерея внутри карточки листинга):** ограничить 2–3 кадрами максимум (не всей галереей товара), с ленивой подгрузкой — 2-й и 3-й кадр запрашиваются только по первому касанию/свайпу внутри карточки, а не при рендере списка (иначе лишние загрузки изображений на весь список из 24 карточек). Свайп внутри карточки использует `touch-action: pan-x` только в зоне фото; вертикальный скролл страницы не должен перехватываться — жест начинает работать как горизонтальный свайп только после явного горизонтального движения пальца (порог ~10px), иначе передаётся странице как вертикальный скролл.
- **PDP**: галерея full-width свайп с точками, блок покупки сразу под галереей (без скролла до цены — это критично, цена должна быть видна в первые 1–2 свайпа), **sticky bottom bar покупки** обязателен (см. 4.8/5.3), таб-аккордеоны вместо горизонтальных табов.
- **Корзина/checkout**: список позиций стек, итоговая панель — sticky снизу с наивысшим z-index над bottom bar навигации (если она есть), крупные тач-таргеты ≥44×44px везде. **Автофокус первого поля — только на desktop** (правка С11, см. 5.5) — на мобильном чекауте автофокус не ставится вовсе, чтобы не вызывать клавиатуру и скачок вьюпорта сразу при открытии страницы.
- **Личный кабинет**: табы вместо боковой навигации, статус заказа — компактный бейдж вместо полной таймлайн-визуализации (таймлайн доступен по тапу «Подробнее о статусе»).
- **Плавающая кнопка звонка**: нижний правый угол, но выше нижнего sticky bar товара (стек друг над другом с отступом 12px, не перекрываются), на страницах без sticky bar — стандартная позиция 16px от края. Меню по клику — «Позвонить / Заказать звонок» (см. правку С12, WhatsApp/Telegram не включены без подтверждения).
- Все интерактивные touch-таргеты — хит-зона ≥44×44px даже при меньшем визуальном размере иконки (правка В8), расстояние между соседними кликабельными элементами ≥8px (согласно CRO-чеклисту).
- Изображения — `srcset`/AVIF/WebP, ленивая загрузка кроме LCP-элемента, `aspect-ratio` фиксирован всегда (защита от CLS).

**3 ключевых дизайн-решения**:
1. **Тёплая светлая основа + единственный «кричащий» терракотовый CTA** (`#B5482F`) — вся палитра нарочно спокойная (бежевый/белый/венге/латунь в тон реальным фото мебели), поэтому кнопка «Заказать» физически не теряется на фоне 101 товара с фото на белом и коричневом дереве.
2. **Пластика референса gigacode переведена в «латунь/янтарь» вместо неона**: те же scroll-reveal/marquee/счётчики/параллакс/свечения, но `gradient-amber` и `gradient-brass-glow` вместо техно-градиентов — визуально это читается как современный, но explicitно мебельный/премиальный сайт, а не IT-лендинг.
3. **Честность вместо давления как принцип самих компонентов, теперь выверенный против реального API**: диапазон цены «от X до Y ₽» вместо выдуманного постатейного пересчёта, прозрачная строка «доставку рассчитает менеджер» вместо мнимой автокалькуляции, бейджи «Хит»/«Новинка»/«Под заказ» только из реальных полей API (`is_bestseller`, `is_new`, `in_stock`), рейтинг только когда `rating != null`, никаких фейковых скидок/счётчиков/гарантийных цифр без подтверждения заказчиком — это прямое следствие аудита дизайн-системы против `api-contract.md` и `products.json`.

### Шрифты и подключение

- **Unbounded** — заголовки (H1–H4, крупная цена PDP). Веса `500;600;700;800`, обычный + italic. Google Fonts, кириллица подтверждена (cyrillic subset присутствует в семействе).
- **Manrope** — весь остальной текст, кнопки, цифры/счётчики. Веса `400;500;600;700`. Google Fonts, кириллица подтверждена.
- Оба шрифта подключаются через `next/font/google`:

```ts
import { Unbounded, Manrope } from 'next/font/google';

const unbounded = Unbounded({
  subsets: ['cyrillic', 'latin'],
  weight: ['400', '500', '600', '700'],
  variable: '--font-unbounded',
  display: 'swap',
});

const manrope = Manrope({
  subsets: ['cyrillic', 'latin'],
  weight: ['400', '500', '600', '700'],
  variable: '--font-manrope',
  display: 'swap',
});
```

- Никакого третьего шрифта (JetBrains Mono исключён, правка С9) — артикул рендерится как `Код: OM-{id}` в Manrope.
- Все анимируемые/пересчитываемые числа — обязательный CSS `font-variant-numeric: tabular-nums` (правка С7 — у Manrope нет табличных цифр по умолчанию).

### Топ-10 обязательных конверсионных элементов (v1, реализуемых на текущем API)

1. **Sticky-панель покупки на PDP** — desktop-полоса при скролле + mobile bottom bar с ценой и кнопкой «Оформить заказ», обрабатывает случай невыбранного размера/цвета (раздел 5.3).
2. **Кнопка «Купить в 1 клик»** на PDP и на каждой карточке листинга — модалка имя+телефон+`product_slug` через `POST /callbacks/`, без предоплаты (разделы 4.10, 4.1).
3. **Телефон, видимый всегда**, включая sticky-компакт хедер (не только topbar) — критично для сегмента 50+ (правка В6, раздел 4.5).
4. **Обратный звонок в нескольких точках**: хедер, PDP, финальный блок главной, плавающая кнопка (раздел 4.10, 5.1 п.13).
5. **Гостевой checkout одной страницей** — имя+телефон обязательны, email опционален, без принудительной регистрации (раздел 5.5).
6. **Честная цена**: «от X ₽» без скрытых доплат, диапазон «от X до Y ₽» с пояснением «зависит от размера и ткани» вместо придуманного пересчёта (правка К2, раздел 5.3).
7. **Cart-drawer при добавлении в корзину** вместо только тоста — показывает состав корзины и ведёт к оформлению без потери контекста (правка С1, раздел 4.11).
8. **Честные бейджи из реальных полей API**: «Хит продаж» (`is_bestseller`), «Новинка» (`is_new`), «Под заказ» (`in_stock=false`) — контраст текста проходит AA (правки В1, К3, К4).
9. **Прозрачная доставка**: «доставим в любой населённый пункт России, точную стоимость и срок сообщит менеджер» + форма «город+телефон → перезвоним с расчётом» вместо мнимой автокалькуляции (правка В10, разделы 5.3/5.8/5.1 п.11).
10. **Все формы ссылаются на реальную страницу политики обработки персональных данных** (правка В12) — обязательное юридическое условие сбора телефонов/email по 152-ФЗ, применяется к чекауту, звонку, 1-клику, отзыву, лид-магниту.
